Video Player App

An Angular 19 + TypeScript app that lists videos from the Pexels Videos API, plays them, and lets you search and pick a resolution — built as a focused, fully-typed MVP.

Angular 19 · Standalone Signals + RxJS Raw SCSS · zero UI libs Jest · jest-preset-angular Strict mode · no any

What was built

Five core features, each wired end-to-end with typed data flow and explicit loading / error / empty states.

🎬

Fetch & list

Loads videos from Pexels and renders them as cards showing poster, author and duration.

▶️

Play

Selecting a card opens a deep-linkable /video/:id route with native HTML5 controls.

🔍

Search (debounced)

Search by author name with a 400ms debounce — filters the loaded pool client-side.

⚙️

Quality selection

A native <select> of renditions sorted by height, shown only when more than one exists.

🚦

Loading / error / empty

A single @switch on status drives skeletons, an error+retry view, the empty state and the grid.

♿

Accessible & fast

Keyboard-navigable, ARIA roles, OnPush everywhere, lazy thumbnails & lazy routes.

Key decisions

Deliberate, minimal choices — modern Angular without extra dependencies.

AreaDecisionWhy
StateAngular Signals + RxJS for the search streamMinimal boilerplate; RxJS where it shines (debounce / cancel).
StylingRaw SCSS, zero UI librariesFull control, no dependencies, fast.
PlayerSeparate route /video/:idDeep-linkable; clean separation from the list.
Auth tokenOptional API key via interceptorEndpoints work unauthenticated; one place to add a key if needed.
TestingJest (jest-preset-angular)Fast unit tests for service, store, pipe and card.
UI languageEnglish onlyAll visible text, labels, alt and aria.
A note on search: Pexels' server-side /search matches by topic/content, not by author — and Pexels videos have no title field (the display title is derived from the URL slug). The chosen UX is search by author, a real field that isn't indexed server-side, so search runs client-side over a loaded pool of ~80 videos. An honest trade-off; server-side topic search remains available on the service.

Project structure

Angular 19, standalone components, Signals and HttpClient. Feature routes are lazy-loaded.

src/
  environments/environment.ts        # base URL + optional pexelsApiKey
  app/
    models/video.model.ts            # Video, VideoFile, PexelsUser, PexelsResponse (no `any`)
    core/
      pexels.service.ts              # getPopular(), search(q), getById(id) — typed HttpClient
      pexels.interceptor.ts          # attaches Authorization header iff a key is set
    features/
      video-list/
        video-list.component.*        # route '' — search box + grid + states
        video-list.store.ts           # signals: videos/status/query; switchMap data flow
      video-player/
        video-player.component.*      # route 'video/:id' — native player + quality select
    shared/
      video-card/video-card.component.*   # presentational card (input.required<Video>)
      duration.pipe.ts                # seconds -> "m:ss"
      video-title.ts                  # derive a display title from the url slug
    app.routes.ts                    # '' (list), 'video/:id' (player), '**' -> ''
    app.config.ts                    # provideRouter + provideHttpClient(withInterceptors)

Cross-cutting quality

🔒 Type safety

All API calls use generics (http.get<PexelsResponse>()); tsconfig runs in strict mode with strictTemplates. No any anywhere.

♿ Accessibility

Search <input type="search"> with a linked label; cards are focusable <a> elements with descriptive alt; native quality <select>; state regions use role="status" / role="alert"; visible :focus-visible outlines.

⚡ Performance

switchMap cancels in-flight requests; every component is ChangeDetectionStrategy.OnPush with Signals; @for uses track video.id; thumbnails are loading="lazy"; both feature routes are code-split with loadComponent.
